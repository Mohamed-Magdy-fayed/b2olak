description: Add/edit a translation key in both en.ts and ar.ts, or audit for i18n gaps
---

Manage bilingual strings for ba2olak. The dictionaries are
`packages/i18n/src/dictionaries/en.ts` and `ar.ts`. The shared `Dictionary` type enforces
parity at typecheck time, so **a key added to one file MUST be added to the other in the
same change** (same nested path, same key order).

Argument: `$ARGUMENTS` — either a key + intent to add/edit (e.g. `orders.cancelConfirm
"Cancel this order?"`), or `audit`.

### Add / edit mode (default)

1. Read both `en.ts` and `ar.ts`. Locate the right nested section (e.g. `common`, `auth`,
   `home`, `orders`, …) — match the existing grouping; don't invent a top-level section
   if a fitting one exists.
2. Add or edit the key in **both** files at the same path and position:
   - `en.ts`: the English copy.
   - `ar.ts`: a natural **Arabic** translation (not transliteration). If unsure of the
     best Arabic wording, propose it and flag it for review — never leave it English.
3. Preserve formatting (the `defineTranslation`/`dt` helper usage, trailing commas, key
   ordering).
4. Run `npm run typecheck --workspace packages/i18n` (or full `npm run typecheck`) to
   confirm parity holds.

### Audit mode (`/i18n audit`)

- Compare `en.ts` vs `ar.ts` and list any keys present in one but missing in the other,
  and any AR values that are suspiciously identical to EN (likely untranslated).
- Grep the apps for likely **hardcoded user-facing strings** — string literals in JSX /
  RN `<Text>` that aren't wrapped in `t(...)`. Report file:line candidates; don't auto-fix
  unless asked.
- Spot-check for directional CSS (`ml-/mr-/pl-/pr-`) and raw color classes while reading,
  since those break RTL / the design system — mention them but defer detail to the
  `i18n-rtl-auditor` subagent for a full sweep.
